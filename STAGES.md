# signals — план по этапам

Работаем последовательно. Каждый этап — самостоятельный проверяемый результат,
не ломающий предыдущий. Ниже статус и куда каждый этап ляжет в архитектуре.

| Этап | Задача | Статус | Где живёт |
|------|--------|--------|-----------|
| 1 | Рефакторинг (новая архитектура, ядро без Qt) | ✅ готово | `signals/plugins`, `domain`, `engine`, `io`, `services`, `instruments`, `contrib` |
| 2 | Полное восстановление исходного функционала (GUI) | ✅ готово | `signals/app_qt` (дерево, графики, сводка, экспорт Excel, сохранение) |
| 3 | Поддержка HANTEK (DLL раздельно x86/x64) | ✅ готово | `signals/contrib/inst_hantek.py` + `signals/contrib/hantek_dll/{x86,x64}` |
| 4 | Модуль автообновлений (GitHub, выбор ветки, замена изменённых файлов) | ✅ готово | `signals/services/updater.py` + `signals/app_qt/update_dialog.py` |
| 5 | Сохранение пользовательских шаблонов обработки/формул | ⏭ следующий | `signals/io/templates.py` + рантайм-реестры COLUMNS/FUNCTIONS |
| 6 | Интерактивность и эргономика (ctrl+2клик по выражению в легенде → датасет в формулу и т.п.) | ⏳ | `signals/app_qt/summary_dialog.py`, виджеты легенды |
| 7 | Документация по созданию модификаций/сценариев | ⏳ | `docs/plugins.md`, `docs/scenarios.md` |
| 8 | Интерактивное обучение в демо при первом запуске | ⏳ | `signals/app_qt/onboarding.py` |
| 9 | Починка всплывших багов | ⏳ | по ходу |

## Что сделано на этапе 1 (рефакторинг)

Ядро приложения без зависимости от Qt — тестируемое и запускаемое из CLI.

**Точки расширения и плагины** (`signals/plugins`, `signals/extpoints.py`)
* Единый реестр для всех расширений: приборы, колонки, функции выражений,
  стратегии поиска фронта, парсеры, экспортёры.
* Автообнаружение из трёх источников: встроенные модули `contrib/`, drop-in
  `.py` в папке плагинов, pip-пакеты через entry points.
* Интроспекция возможностей: прибор сам сообщает `capabilities()`, потребитель
  спрашивает `supports(obj, Cap.GENERATOR)` — без списков моделей.

**Доменная модель** (`signals/domain.py`)
* Типизированные `Channel / MeasurementParams / Analysis / Subject / Project`
  вместо словарей с магическими ключами. Пользовательские метки — поля модели.
* `Analysis.dirty` и `Project.dirty_analyses()` — основа инкрементального сейва.

**Движок** (`signals/engine.py`) — чистый расчёт АЧХ + временные ряды
(сырой/сглаженный/обрезанный); стратегия поиска фронта берётся из реестра.

**Ввод-вывод** (`signals/io`)
* Парсеры форматов оригинала (CSV новый/старый, Excel) — открываются старые данные.
* Хранилище проекта: папка `project.json` + `data/<id>.csv`; запись атомарная
  (temp + `os.replace`), сохранение инкрементальное (только изменённые анализы),
  упаковка в один `.sigproj` (zip) для передачи/сжатия. Pickle убран.

**Сервисы** (`signals/services`)
* `analyze_full` — движок + значения всех колонок (вкл. пользовательские).
* `MeasurementService` — свип+захват для прибора с генератором (Hantek).

**Приборы** (`signals/instruments.py`, `signals/contrib/inst_*`)
* Протоколы `Oscilloscope` / `Generator`; Hantek (ctypes-биндинг из заголовков
  SDK, со встроенным DDS) и регистрации Tektronix/Rigol.

### Проверка этапа 1
```bash
cd signals_kernel
python -m examples.demo          # автообнаружение, capabilities, рантайм-колонка, расчёт АЧХ
```
Тест конвейера данных (парсинг → анализ → метрики → сохранение/загрузка → упаковка)
проходит на сгенерированном CSV. GUI появится на этапе 2.

### Чего на этапе 1 намеренно НЕТ
* Графического интерфейса (это этап 2).
* Боевой проверки Hantek на железе (биндинг написан из заголовков SDK; финальную
  сверку упаковки структур и тайминга свипа делаем на этапе 3 с прибором).

## Паритет UI с оригиналом (этап 2)

Проверено, что в переработанном интерфейсе функций не меньше, чем было:

| Функция оригинала | В переработке |
|-------------------|---------------|
| Добавить предмет | ✅ панель «Управление» |
| Добавить измерения (в активный предмет) | ✅ в выбранный предмет или по имени файла |
| Загрузить анализ / открыть проект | ✅ «Открыть проект» (.sigproj/папка) |
| Сохранить всё | ✅ «Сохранить всё» (инкрементально) |
| Сохранить выбранные (отмеченные) | ✅ «Сохранить выбранные» → .sigproj |
| Настройки столбцов (показ/скрытие/удаление) | ✅ окно «Столбцы» |
| Фильтры по столбцам + сброс | ✅ панель фильтра + «Сбросить фильтры» |
| Сводка по отмеченным | ✅ «Сводка» |
| Перемещение анализа между предметами | ✅ контекстное меню «Переместить в предмет» |
| Удаление (Delete / кнопка) | ✅ кнопка + Delete + контекстное меню |
| Переименование | ✅ инлайн в дереве (F2) |
| Графики и подстройка (3 графика, строб, маркеры, прогноз) | ✅ окно «Настройка измерения» |
| Прибор: поиск, измерение, чтение, периодическое чтение | ✅ панель «Прибор» |
| Запрос сохранения при выходе | ✅ closeEvent |
| Динамические столбцы-метрики | ✅ из реестра COLUMNS |

Сводный диалог и менеджер приборов восстановлены по основным функциям; их более
тонкие возможности (расширенная алгебра датасетов, выбор пары генератор+осцилл.)
дорабатываются итеративно вместе с этапами 5–6.
